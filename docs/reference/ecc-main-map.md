## 1. Repository Overview

ECC ("Everything Claude Code", also published on npm as `ecc-universal`) is a large, harness-native agent operating system distributed as a Claude Code plugin and marketplace. The repository (VERSION `2.0.0`, MIT, by Affaan Mustafa) bills itself as "the harness-native operator system for agentic work" — not just config files but a complete system of agents, skills, hooks, rules, MCP conventions, install tooling, and an operator control plane, evolved over 10+ months of daily product engineering. A central design goal is cross-harness portability: the same skill/agent/rule catalogue is mirrored to run on Claude Code, Codex, Cursor, OpenCode, Gemini, Zed, Qwen, Kiro, Trae, CodeBuddy and more.

### Philosophy

`SOUL.md`, `AGENTS.md`, and `RULES.md` encode five core principles: Agent-First (route work to specialists early), Test-Driven (tests before trusting implementation, 80%+ coverage), Security-First (validate inputs, protect secrets, safe defaults), Immutability (explicit state transitions over mutation), and Plan-Before-Execute (decompose complex changes into phases). Every rules file and `CLAUDE.md` also carries a "Prompt Defense Baseline" treating fetched/untrusted content as suspicious. The stated headline scale is 67 agents, 271 skills, and 92 (now-legacy) command shims, though individual docs quote older counts — the catalogue is large and churns weekly.

### Repository organization

The canonical workflow surface is `skills/` (~271 skill folders, each `skills/<name>/SKILL.md` with YAML frontmatter). `agents/` holds ~67 specialist subagents (planner, code-reviewer, security-reviewer, language build-resolvers/reviewers, loop-operator, harness-optimizer). `commands/` (~92 slash commands) is now a legacy compatibility surface — new work lands in skills first. `rules/` holds always-follow common + per-language guidance, `hooks/` holds the event-driven automation graph, and `scripts/` (~200 Node.js files) provides cross-platform hook helpers plus the `ecc`/`ecc-install`/`ecc-control-pane` CLIs. Supporting directories (`manifests/`, `config/`, `schemas/`, `scaffolds/`, `contexts/`, `mcp-configs/`, `plugins/`, `integrations/`, `src/`, `ecc2/`, `examples/`, `legacy-command-shims/`, `assets/`, `docs/`) and the multi-tool mirror dotfolders round out the tree.

### Install model

Onboarding is layered: **modules** (`manifests/install-modules.json`) are the atomic unit — each names a `kind`, a path list, supported harness `targets`, dependencies, cost, and stability (e.g. `rules-core`, `agents-core`, `hooks-runtime`, `framework-language`, `database`, `security`, `orchestration`). **Components** (`manifests/install-components.json`) group modules into installable families (`baseline:*`, `lang:*`). **Profiles** (`manifests/install-profiles.json`) are named bundles of modules: `minimal`, `opencode`, `core`, `developer` (the default), `security`, `research`, and `full`. Installation runs via `install.sh`/`install.ps1` (or the `ecc-install` bin) with `--target <harness> --modules <ids>`, which resolves repo paths and rewrites hook commands against the user's actual config root. The `/project-init` skill detects a project's stack via `config/project-stack-mappings.json` (indicator files → rules, skills, default build/test/lint/format commands, and allow/deny permissions) and produces a dry-run onboarding plan; `schemas/` provides JSON Schemas validating every manifest, hook graph, and install-state file in CI.

#### Workflows & Orchestration

ECC composes multi-step and multi-agent work through several skill/command families that all delegate to existing agents rather than re-implementing work. The **orch-\*** family (`orch-add-feature`, `orch-change-feature`, `orch-fix-defect`, `orch-refine-code`, `orch-build-mvp`) are thin wrappers over a shared engine, `orch-pipeline`, which defines a gated Research → Plan → TDD → Review → Commit pipeline, a size classifier (trivial/small/standard/large that scales ceremony to blast radius) and two human gates, delegating each phase to `/feature-dev`, `/plan`, `/code-review`, `tdd-workflow`, `/gan-build`, etc. The **multi-\*** family (`multi-plan`, `multi-execute`, `multi-backend`, `multi-frontend`, `multi-workflow`) runs multi-model workflows (research/plan/execute/optimize/review) while keeping Claude as the only filesystem writer. The **gan-\*** harness (`gan-style-harness`, `/gan-build`, `/gan-design`) separates a Generator agent from a ruthlessly strict Evaluator in a bounded scored loop (inspired by Anthropic's long-running-harness work) to drive quality past single-agent self-review. `santa-method` (`/santa-loop`) is an adversarial dual-review convergence loop where two independent reviewers must both approve before shipping. `plan-orchestrate` reads a plan doc and emits ready-to-paste `/orchestrate custom` agent-chain prompts. `team-agent-orchestration` manages agent squads via work items, an agent Kanban, branch/worktree isolation, a control pane, and merge gates. `autonomous-loops` / `continuous-agent-loop` (with `/loop-start`, `/loop-status`, the loop-operator agent) provide sequential/infinite/PR/RFC-DAG loop patterns with quality gates and recovery controls, and `dynamic-workflow-mode` designs task-local harnesses with eval gates. These integrate with the **hooks** runtime (lifecycle observation, quality gates, GateGuard, cost/context monitoring — see the hooks section) and with **`.github/workflows`** CI: `ci.yml` runs the validate/test matrix (3 OSes × 3 Node versions × 4 package managers) executing `scripts/ci/*` validators for agents, commands, rules, skills, hooks, install manifests, unicode safety, and personal-path leakage, alongside reusable validate/test/release workflows, `supply-chain-watch.yml`, and `monthly-metrics.yml`. Worktree/tmux orchestration runtime lives in `scripts/orchestrate-worktrees.js` + `scripts/lib/tmux-worktree-orchestrator.js` (the `orchestration` install module), and the Rust `ecc2/` control plane is the emerging "layer above" that manages many sessions at once.
